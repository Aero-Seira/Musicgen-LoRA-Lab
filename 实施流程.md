## 2. 总体实施策略

采用三阶段推进：

```text
阶段 A：跑通最小闭环
  数据预处理 -> 预训练基线生成 -> 客观评估 -> 听测表

阶段 B：补齐 LoRA 实验
  确定可用 LoRA 训练实现 -> 小样本训练 -> LoRA 生成 -> 对比评估

阶段 C：完成泛化和报告
  外部音频测试 -> 主观听测 -> 图表整理 -> 报告撰写
```

不要一开始就把重点放在完整 LoRA 训练上。先保证预训练基线、评估和报告材料能稳定产出，再推进 LoRA。

## 3. 阶段 A：跑通最小闭环

### 3.1 环境准备

本项目使用 `uv` 管理环境和依赖。算力平台上先安装 `uv`：

```bash
curl -LsSf https://astral.sh/uv/install.sh | sh
```

然后同步依赖：

```bash
uv sync
```

默认 Python 版本写在 `.python-version` 中，为 `3.10`。如平台需要使用 3.11，可以：

```bash
PYTHON_VERSION=3.11 ./start.sh smoke
```

如果 `audiocraft` 安装失败，优先按 AudioCraft 官方说明安装匹配版本的 PyTorch 和 CUDA。

### 3.2 数据预处理

运行：

```bash
python scripts/preprocess_gtzan.py
```

预期输出：

```text
data/processed/train/
data/processed/val/
data/processed/test/
```

预期数量：

```text
train: 700
val: 150
test: 150
```

验收标准：

- 每条音频为 `30s`
- 采样率为 `32000Hz`
- 单声道
- `data/processed/test/` 中每个流派都有样本

### 3.3 预训练基线生成逻辑

`generate_baseline.py` 已使用 MusicGen 的续写接口：

```python
model.set_generation_params(duration=30)
model.generate_continuation(prompt_20s, prompt_sample_rate, descriptions=[""], progress=True)
```

注意：20 秒 prompt 续写 10 秒时，MusicGen 的生成时长按 `30s` 设置，然后脚本只截取后 `10s` 保存为 `{原始stem}_baseline.wav`。

### 3.4 小样本跑基线

先不要全量跑，先每个流派 1 条：

```bash
python scripts/generate_baseline.py --samples-per-genre 1
```

验收标准：

- `outputs/baseline/{genre}/` 下生成 `.wav`
- 每条生成音频约 `10s`
- 音频能正常播放

通过后再扩大：

```bash
python scripts/generate_baseline.py --samples-per-genre 10
```

### 3.5 客观评估逻辑

`evaluate_audio.py` 已保证比较对象正确：

- condition：原始音频 `0s-20s`
- real target：原始音频 `20s-30s`
- generated：模型生成的 `10s`

脚本同时支持 `*_baseline.wav` 和 `*_lora.wav`，并额外输出逐样本 CSV。

### 3.6 跑基线评估

运行：

```bash
python scripts/evaluate_audio.py \
  --generated-dirs outputs/baseline \
  --output outputs/metrics/baseline_results.json
```

验收标准：

- 能输出每个流派的指标
- `outputs/metrics/baseline_results.json` 存在
- 指标包括：
  - `mel_mse`
  - `chroma_sim`
  - `smoothness`

完成阶段 A 后，项目已经有可交付的基线实验。

## 4. 阶段 B：补齐 LoRA 实验

### 4.1 LoRA 技术路线

当前 `train_lora.py` 采用 AudioCraft + PEFT 路线：

1. AudioCraft compression model 将 30 秒音频编码为 EnCodec tokens。
2. 对 MusicGen 的 LM 部分应用 PEFT LoRA。
3. 使用 token cross entropy 训练 LoRA 参数。
4. 保存 adapter 到 `outputs/lora/lora_adapter/`。

这一路线用于资源受限条件下的风格适配。若算力平台上 AudioCraft 内部接口版本不兼容，再切换到社区 MusicGen LoRA/DreamBooth 脚本。

### 4.2 小样本 LoRA 训练

先只使用少量样本：

- 每个流派 `5-10` 条
- batch size `1`
- epoch `1-3`
- rank `8`
- learning rate `1e-4` 或 `5e-5`

验收标准：

- 训练能启动
- loss 能正常记录
- adapter 能保存
- 能加载 adapter 生成音频

### 4.3 LoRA 生成

小样本训练通过后，生成测试集对比结果：

```bash
python scripts/generate_lora.py \
  --adapter-path outputs/lora/lora_adapter \
  --samples-per-genre 10
```

输出：

```text
outputs/lora/{genre}/{stem}_lora.wav
```

验收标准：

- 和 baseline 使用同一批 test 样本
- 文件名 stem 能与 baseline 对齐
- 音频约 `10s`

### 4.4 LoRA 对比评估

运行：

```bash
python scripts/evaluate_audio.py \
  --generated-dirs outputs/baseline outputs/lora \
  --output outputs/metrics/baseline_vs_lora.json
```

对比重点：

- LoRA 的 `mel_mse` 是否降低
- LoRA 的 `chroma_sim` 是否提高
- LoRA 的 `smoothness` 是否更小
- 主观听感是否更接近原始流派

## 5. 阶段 C：外部泛化测试

### 5.1 准备外部音频

放入：

```text
data/raw/external/
```

建议准备 `10-20` 条：

- GTZAN 相近风格音乐
- 自录哼唱
- 简单乐器片段
- 少量环境声或非典型音频作为鲁棒性测试

要求：

- `.wav`
- 至少 `20s`
- 最好裁剪到 `30s`

### 5.2 外部音频生成

使用 LoRA 生成脚本：

```bash
python scripts/generate_lora.py \
  --adapter-path outputs/lora/lora_adapter \
  --external-dir data/raw/external \
  --output-external outputs/external
```

输出：

```text
outputs/external/{stem}_lora.wav
```

### 5.3 外部音频分析

外部音频通常没有严格真实后 `10s` 对照，因此重点使用：

- 主观听测
- 过渡平滑性
- 案例分析
- 成功/失败样本对比

报告中应把外部音频作为“泛化性实验”，不要和 GTZAN 测试集的客观指标完全等价。

## 6. 主观听测流程

### 6.1 生成听测表

当 baseline 和 lora 都生成后，运行：

```bash
python scripts/make_listening_test.py \
  --baseline-dir outputs/baseline \
  --lora-dir outputs/lora \
  --output reports/listening_test.html \
  --n-samples 10
```

### 6.2 听测设计

建议邀请 `5-10` 名同学评分。

评分维度：

- 连贯性：1-5
- 风格一致性：1-5
- 音质：1-5
- 偏好：baseline 或 lora

注意：当前 HTML 虽然命名为 A/B，但 A 固定是 baseline，B 固定是 lora。若要严格盲听，需要进一步随机 A/B 顺序，并保存映射表。

## 8. 最终实施顺序

建议严格按以下顺序执行：

```text
1. 确认环境和依赖
2. 运行 `preprocess_gtzan.py`
3. 每流派 1 条样本跑 baseline
4. 跑 baseline 客观评估
5. 每流派 10 条样本跑 baseline
6. 小样本 LoRA 训练
7. LoRA 生成同一批测试样本
8. baseline vs lora 客观评估
9. 准备外部音频并做泛化生成
10. 生成主观听测表并收集评分
11. 汇总图表和结果
12. 撰写最终报告
```

## 9. 上平台推荐命令

先跑 smoke test：

```bash
./start.sh smoke
```

在 5090 32G 上，`smoke` 和 `full` 默认都会使用 `facebook/musicgen-melody`。`smoke` 只是减少样本数和 epoch，用于验证环境。

`uv.lock` 已按 5090/Blackwell 风险调整到 PyTorch 2.8 + CUDA 12.8 相关依赖，避免旧 PyTorch 2.1/cu121 在 5090 上不兼容。

确认无误后跑正式实验：

```bash
./start.sh full
```

如果只想先产出 baseline：

```bash
./start.sh baseline
```

需要覆盖默认参数时使用环境变量：

```bash
MODEL_NAME=facebook/musicgen-small SAMPLES_PER_GENRE=5 EPOCHS=2 ./start.sh full
```

OpenBayes 数据绑定到 `/openbayes/input/input0` 时，脚本会自动识别。也可以显式指定：

```bash
RAW_GTZAN_DIR=/openbayes/input/input0 ./start.sh full
```
