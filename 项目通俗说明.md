# MusicGen-LoRA-Lab 项目通俗说明

## 1. 这个项目是做什么的

这个项目研究的是“音乐续写”。

简单说，就是给模型一段音乐的前 20 秒，让模型根据这 20 秒继续生成后面的 10 秒。我们把模型生成的 10 秒和原始音乐真实的后 10 秒做对比，看看生成结果是否更接近真实音乐、风格是否更一致、衔接是否更自然。

项目里比较了两种方式：

| 方式 | 含义 |
| --- | --- |
| 预训练基线 | 直接使用 Meta 发布的 `facebook/musicgen-melody` 模型，不做额外训练 |
| LoRA 微调 | 在 `facebook/musicgen-melody` 的基础上，用 GTZAN 数据集训练一个小的 LoRA adapter |

这里的 LoRA 可以理解为“给大模型外挂一个小补丁”。它不会重新训练整个 MusicGen 大模型，而是只训练少量额外参数。这样训练成本更低，保存的模型文件也更小。

## 2. 这个项目最终能做到什么

项目目前能完成以下事情：

1. 自动整理 GTZAN 音乐数据集。
2. 使用 MusicGen-Melody 根据前 20 秒音乐生成后 10 秒续写。
3. 对 MusicGen-Melody 做 LoRA 微调。
4. 保存 LoRA adapter，后续可以单独加载使用。
5. 用已训练好的 LoRA adapter 对测试集或外部音乐做续写推理。
6. 对 baseline 和 LoRA 生成结果做客观指标测评。
7. 生成 A/B 盲听网页，方便小组成员做人耳主观评价。
8. 输出实验报告需要的结果文件、指标文件和说明文档。

项目不直接做到以下事情：

- 不训练一个完整的 MusicGen 大模型。
- 不保证 LoRA 在所有音乐风格上都比原模型更好。
- 不自动完成多人听测统计，需要人工听音频并评分。
- 不提供网页服务或 API 服务，只提供命令行推理脚本。

## 3. 项目整体流程

完整实验可以理解为一条流水线：

```text
GTZAN 原始音乐
    ↓
数据预处理
    ↓
切分 train / val / test
    ↓
预训练模型生成 baseline
    ↓
LoRA 微调
    ↓
LoRA 模型生成续写
    ↓
客观指标测评
    ↓
生成听测页面和报告材料
```

其中：

- `train` 用来训练 LoRA。
- `val` 用来观察训练过程中是否过拟合。
- `test` 用来比较 baseline 和 LoRA 的生成效果。

每条 GTZAN 音频默认长度为 30 秒。项目使用：

```text
0-20 秒：作为模型输入
20-30 秒：作为真实答案
```

模型生成时也会输出 10 秒续写，然后和真实的 20-30 秒片段比较。

## 4. 每个重要文件和目录在做什么

### 4.1 根目录文件

| 文件 | 作用 |
| --- | --- |
| `README.md` | 快速启动说明，适合已经知道项目要怎么跑的人 |
| `项目通俗说明.md` | 当前这份文档，适合先理解项目整体 |
| `MusicGen-LoRA-完整方案.md` | 更偏研究方案和作业规划 |

### 4.2 `scripts/` 目录

这个目录是项目的核心代码。

| 脚本 | 做什么 |
| --- | --- |
| `preprocess_gtzan.py` | 预处理 GTZAN，统一采样率、长度，划分训练/验证/测试集 |
| `generate_baseline.py` | 使用原始 MusicGen-Melody 生成 baseline 续写 |
| `train_lora.py` | 对 MusicGen-Melody 的语言模型部分进行 LoRA 微调 |
| `generate_lora.py` | 加载基座模型和 LoRA adapter，生成 LoRA 续写 |
| `evaluate_audio.py` | 计算生成音频和真实音频之间的客观指标 |
| `make_listening_test.py` | 生成 A/B 主观听测 HTML 页面 |
| `run_platform.sh` | 被 `start.sh` 调用的完整平台流水线 |

### 4.3 `data/` 目录

`data/` 用来放数据。

```text
data/
├── raw/
│   ├── gtzan/       # 本地 GTZAN 原始数据
│   └── external/    # 外部测试音乐
└── processed/
    ├── train/       # 训练集
    ├── val/         # 验证集
    └── test/        # 测试集
```

在 OpenBayes 平台上，GTZAN 通常不需要复制到项目目录，数据会绑定在：

```text
/openbayes/input/input0
```

脚本会优先识别这个路径。

### 4.4 `outputs/` 目录

`outputs/` 是实验输出目录。

```text
outputs/
├── baseline/     # 完整训练流水线中的 baseline 生成结果
├── lora/         # LoRA 生成结果、训练日志、adapter 权重
├── infer/        # 一键推理脚本生成的新结果
├── external/     # 外部音频生成结果
└── metrics/      # 客观指标结果
```

最重要的是：

```text
outputs/lora/lora_adapter/
```

这里保存的是训练好的 LoRA adapter。里面通常包括：

| 文件 | 作用 |
| --- | --- |
| `adapter_model.safetensors` | LoRA 权重 |
| `adapter_config.json` | LoRA 配置 |
| `README.md` | adapter 说明 |

这个 adapter 不能单独使用，必须配合基座模型：

```text
facebook/musicgen-melody
```

### 4.5 `reports/` 目录

`reports/` 用来放审计、报告和听测材料。

| 文件 | 作用 |
| --- | --- |
| `output_audit.md` | 对已有 `outputs/` 的审计总结 |

## 5. 怎么安装环境

项目推荐使用 `uv` 管理环境。

先安装 `uv`：

```bash
curl -LsSf https://astral.sh/uv/install.sh | sh
```

然后在项目根目录同步依赖：

```bash
uv sync --locked
```

在 OpenBayes 或 Linux GPU 平台上，也可以指定 Python 版本：

```bash
uv sync --python 3.10 --locked
```

本项目已经针对 5090 32G 平台做过依赖约束，默认使用：

```text
facebook/musicgen-melody
```

如果显存不足，再考虑降级到：

```text
facebook/musicgen-small
```

## 6. 怎么运行完整实验

如果是第一次在算力平台上完整跑训练和实验，推荐先跑小样本测试：

```bash
./start.sh smoke
```

这个模式会少量采样，主要用来确认：

- Python 环境能否创建。
- 依赖能否安装。
- MusicGen 能否加载。
- 数据能否读取。
- LoRA 训练和生成接口能否跑通。

确认没问题后，再跑完整实验：

```bash
./start.sh full
```

完整实验会执行：

1. 预处理 GTZAN。
2. 生成 baseline。
3. 测评 baseline。
4. 训练 LoRA。
5. 使用 LoRA 生成结果。
6. 对比 baseline 和 LoRA。
7. 生成听测页面。

如果只想生成 baseline，不训练 LoRA：

```bash
./start.sh baseline
```

常用参数覆盖：

```bash
SAMPLES_PER_GENRE=5 EPOCHS=2 ./start.sh full
MODEL_NAME=facebook/musicgen-small ./start.sh full
RAW_GTZAN_DIR=/openbayes/input/input0 ./start.sh full
```

## 7. 怎么只做推理和测评

如果已经有训练好的 LoRA adapter，不想重新训练，可以使用：

```bash
./infer_eval.sh compare
```

这个脚本适合“复验结果”或“部署演示”。它默认会：

1. 检查并同步 `uv` 环境。
2. 如果没有处理好的测试集，就自动预处理 GTZAN。
3. 生成 baseline 结果到 `outputs/infer/baseline/`。
4. 生成 LoRA 结果到 `outputs/infer/lora/`。
5. 测评结果写到 `outputs/metrics/infer_eval.json` 和 `outputs/metrics/infer_eval.csv`。
6. 听测页面写到 `reports/infer_listening_test.html`。

常用模式：

```bash
./infer_eval.sh compare    # baseline + LoRA 推理 + 测评
./infer_eval.sh lora       # 只跑 LoRA 推理和测评
./infer_eval.sh baseline   # 只跑 baseline 推理和测评
./infer_eval.sh evaluate   # 只评估已有推理结果
./infer_eval.sh external   # 对外部 wav 做 LoRA 推理
```

如果要指定 GPU：

```bash
DEVICE=cuda ./infer_eval.sh compare
```

如果只想每个流派生成 3 条：

```bash
SAMPLES_PER_GENRE=3 ./infer_eval.sh compare
```

如果要评估已经生成好的正式输出：

```bash
BASELINE_OUTPUT_DIR=outputs/baseline \
LORA_OUTPUT_DIR=outputs/lora \
./infer_eval.sh evaluate
```

## 8. 怎么用 LoRA 模型对自己的音乐做续写

把自己的 wav 文件放到：

```text
data/raw/external/
```

建议每条音频不少于 20 秒。因为项目默认使用前 20 秒作为输入提示。

然后运行：

```bash
./infer_eval.sh external
```

生成结果会在：

```text
outputs/infer/external/
```

也可以手动指定输入输出目录：

```bash
EXTERNAL_DIR=data/raw/external \
EXTERNAL_OUTPUT_DIR=outputs/demo \
./infer_eval.sh external
```

外部音频推理只生成音乐，不会自动计算 GTZAN 指标。原因是外部音频没有标准的“真实后 10 秒答案”，无法像 GTZAN 测试集一样计算 Mel MSE 等对比指标。

## 9. LoRA 模型部署时到底怎么加载

LoRA 部署需要两部分：

```text
基座模型：facebook/musicgen-melody
LoRA adapter：outputs/lora/lora_adapter/
```

Python 加载方式大致是：

```python
from pathlib import Path

from audiocraft.models import MusicGen
from peft import PeftModel

device = "cuda"
model_name = "facebook/musicgen-melody"
adapter_path = Path("outputs/lora/lora_adapter")

model = MusicGen.get_pretrained(model_name, device=device)
model.lm = PeftModel.from_pretrained(model.lm, str(adapter_path))
model.lm.to(device)
model.lm.eval()
```

实际使用时，建议直接复用项目里的：

```text
scripts/generate_lora.py
```

因为这个脚本已经处理了：

- 自动选择 `cuda`、`mps` 或 `cpu`。
- 加载 MusicGen 基座模型。
- 加载 LoRA adapter。
- 修复 PEFT 包装后 AudioCraft 方法可能被隐藏的问题。
- 自动读取 wav。
- 自动重采样。
- 截取前 20 秒作为 prompt。
- 只保存最后 10 秒续写结果。

## 10. 评估指标是什么意思

项目主要使用三个客观指标。

### 10.1 Mel MSE

Mel MSE 比较生成音频和真实后续音频在 Mel 频谱上的差异。

可以粗略理解为：声音整体频谱形状像不像。

```text
越低越好
```

但它不能完全代表人的听感，因为两个音乐片段即使听起来都合理，也可能和真实答案不一样。

### 10.2 Chroma Similarity

Chroma Similarity 比较音高类别和和声相关特征。

可以粗略理解为：和弦、音高走向、调性特征是否接近。

```text
越高越好
```

### 10.3 Smoothness

Smoothness 衡量输入音乐结尾和生成音乐开头之间的频谱变化。

可以粗略理解为：20 秒处接得是否突兀。

```text
越低越好
```

这个指标只反映衔接处的一部分平滑程度，也不能完全替代人工听感。

## 11. 本项目当前实验结果能说明什么

根据已有输出审计，本项目已经完成：

- baseline 生成：100 条 wav。
- LoRA 生成：100 条 wav。
- 每个流派 10 条，共 10 个流派。
- 所有生成音频都是 mono、32000 Hz、10 秒。
- LoRA adapter 已成功保存。
- 客观指标 JSON/CSV 已生成。

已有结果显示：

| 指标 | Baseline | LoRA | 结论 |
| --- | ---: | ---: | --- |
| Mel MSE | 142.6694 | 130.8578 | LoRA 整体更低，约改善 8.28% |
| Chroma Similarity | 0.7061 | 0.7002 | Baseline 略高 |
| Smoothness | 136.1970 | 139.1970 | Baseline 略好 |

比较稳妥的结论是：

> LoRA 微调让生成音频在整体 Mel 频谱距离上更接近真实后续片段，但这种提升不是所有流派都稳定存在，也没有明显改善和声相似度与衔接平滑度。

表现较好的流派包括：

- jazz
- reggae
- disco
- country
- pop

表现较弱的流派包括：

- classical
- rock
- blues
- metal
- hiphop

因此，报告里不要写“LoRA 全面优于原模型”。更严谨的说法是：“LoRA 在部分指标和部分流派上有效，但仍需要进一步调参和主观听测。”

## 13. 常见问题

### 13.1 为什么要先跑 `smoke`

因为 MusicGen 和 AudioCraft 依赖比较重，GPU 平台环境也容易出现 CUDA、PyTorch、xformers 版本问题。`smoke` 用很少样本快速检查流程，能节省排错成本。

### 13.2 为什么有些 GTZAN 音频会跳过

GTZAN 数据集中可能存在损坏文件，例如某些 wav 文件无法被音频库正常解码。项目会在预处理阶段跳过坏文件，并记录到：

```text
data/processed/skipped_bad_audio.csv
```

这比中途报错停止整个实验更稳妥。

### 13.3 为什么生成只保存 10 秒

任务设计是“前 20 秒续写后 10 秒”。MusicGen 内部生成时会生成 30 秒左右的完整片段，脚本最后只截取后 10 秒作为模型续写结果。

### 13.4 为什么外部音频不计算指标

客观指标需要“真实答案”。GTZAN 测试集的真实后 10 秒可以作为答案，但外部音频通常只是待续写素材，没有标准答案，所以只能生成音频，不能自动做同样的对比测评。

### 13.5 为什么 adapter 文件很小

LoRA 只保存少量低秩增量参数，不保存完整 MusicGen 模型。所以 `outputs/lora/lora_adapter/` 大约几十 MB，而不是几个 GB。

### 13.6 为什么不能只拿 adapter 推理

adapter 只是“补丁”，不是完整模型。推理时仍然需要先加载：

```text
facebook/musicgen-melody
```

然后把 adapter 挂到模型的语言模型部分。

## 16. 一句话总结

这个项目用 MusicGen-Melody 做音乐续写，并用 LoRA 在 GTZAN 数据集上进行轻量微调。它可以自动完成数据预处理、baseline 生成、LoRA 训练、LoRA 推理、客观测评和听测页面生成。当前结果说明 LoRA 在整体 Mel 频谱误差上有一定改善，但效果具有流派差异，仍需要主观听测和进一步调参来支撑更强结论。
